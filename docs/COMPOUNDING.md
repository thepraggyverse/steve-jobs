# Compounding With SJ Skills

This plugin borrows one important idea from compound engineering: each use should make the next use easier.

For this project, that means a Jobs-style critique should not end as a clever answer in a chat. It should improve the artifact, sharpen taste, and leave behind a reusable lesson.

## The SJ Compound Loop

| Phase | Question | Useful Skills | Artifact |
|---|---|---|---|
| Sense | What is the real product, story, people, strategy, or learning problem? | `$sj-core-catalog` | Skill route |
| Ground | What artifact or evidence are we judging? | `$sj-product-concrete-artifact-review`, `$sj-product-raw-work-no-filters` | Screen, draft, plan, candidate packet, metrics |
| Simplify | What is the one thing that matters? | `$sj-product-simplify-to-one`, `$sj-product-hack-away-unessential` | Clearer product/job/message |
| Raise The Bar | Where is the standard too low? | `$sj-product-taste-review`, `$sj-people-a-player-bar`, `$sj-product-back-of-cabinet-quality` | Quality critique |
| Ship The Next Artifact | What should change next? | Any domain skill | Revised artifact or next demo |
| Compound | What should the system remember? | `$sj-core-compound-learning`, `$sj-learning-progressive-summarization` | Reusable lesson |

## When To Use The Loop

| Situation | Start With | Then Use | End With |
|---|---|---|---|
| Fuzzy product idea | `$sj-core-catalog` | `$sj-product-customer-backwards`, `$sj-product-simplify-to-one` | `$sj-core-compound-learning` |
| Bloated MVP | `$sj-product-simplify-to-one` | `$sj-product-hack-away-unessential`, `$sj-story-sell-the-improvement` | Revised MVP promise |
| Rough launch | `$sj-story-three-act-launch` | `$sj-story-demo-rehearsal-loop`, `$sj-story-number-context` | Launch script |
| Hiring decision | `$sj-people-a-player-bar` | `$sj-people-pushback-interview`, `$sj-people-missionaries-not-mercenaries` | Bar and interview plan |
| Failed project | `$sj-strategy-failure-apprenticeship` | `$sj-anti-revenge-motive-check`, `$sj-strategy-pivot-to-core` | Failure lesson and next move |
| Founder drift | `$sj-people-truth-to-founder` | `$sj-strategy-startup-discipline`, `$sj-anti-too-much-money-check` | Reality channel and correction |

## Beliefs To Let Go

| Old Belief | SJ Plugin Replacement |
|---|---|
| A Jobs lens means imitating Steve's tone. | Use the skills to produce clearer artifacts, not personality theater. |
| The answer is the output. | The improved product, plan, story, or decision is the output. |
| Taste lives in the human reviewer. | Extract taste into reusable skills, references, prompts, and checklists. |
| More ideas means more creativity. | Better selection and simplification creates more force. |
| Failure review is about comfort. | Failure review should expose facts, false stories, and discipline leaks. |
| One great prompt is enough. | Repeated loops plus captured learning make the system better. |

## Beliefs To Adopt

| New Belief | Practice |
|---|---|
| Taste should be explicit. | Use `$sj-core-compound-learning` after a good review to capture what worked. |
| Plans and stories are artifacts. | Review them with the same seriousness as code or UI. |
| Simplicity is a forcing function. | Use `$sj-product-simplify-to-one` before expanding scope. |
| Reality should reach the decision maker. | Use `$sj-people-truth-to-founder` when metrics, demos, or feedback are filtered. |
| Unseen quality matters. | Use `$sj-product-back-of-cabinet-quality` for internal craft and architecture. |
| The next pass should be easier. | Save reusable prompts, patterns, and failure checks. |

## Maturity Stages

| Stage | Behavior | What To Practice |
|---|---|---|
| 0. Ad hoc prompting | Ask for a Jobs-style answer once in a while. | Start using `$sj-core-catalog`. |
| 1. Single-skill use | Invoke one skill for one problem. | Use exact skill names such as `$sj-product-simplify-to-one`. |
| 2. Skill chains | Route through two or three skills. | Chain product, story, and anti-pattern checks. |
| 3. Artifact loops | Apply skills to real screens, plans, copy, metrics, and people decisions. | Use concrete artifact review and revise the thing. |
| 4. Compounding | Capture lessons into prompts, docs, references, or new skills. | End with `$sj-core-compound-learning`. |
| 5. Taste system | Your team has stable patterns for product craft, story, hiring, and failure review. | Refresh references and prune stale patterns. |

## Three Questions Without Tooling

If you do not know which skill to invoke, ask:

1. What is the hardest judgment call here?
2. What did we reject, and why?
3. What are we least confident about?

Then route the answer through `$sj-core-catalog`.

## What To Capture

Capture lessons that would help a future agent or future version of you.

| Capture | Example |
|---|---|
| Product taste | "When onboarding needs three paragraphs, the flow is probably wrong." |
| Story pattern | "Lead with the pain, then show the product as relief." |
| Hiring bar | "A-player evidence must include shipped work and intelligent pushback." |
| Strategy warning | "Funding can hide lack of business clarity." |
| Failure pattern | "The first story we tell after failure is often self-protective." |
| Better prompt | "Use `$sj-product-customer-backwards` before discussing implementation." |

## What Not To Capture

- Long source quotations
- Generic inspiration
- One-off personal preferences that will not matter later
- Notes that flatter the decision instead of making it clearer
- Anything private or copyrighted that should not live in a public repo
